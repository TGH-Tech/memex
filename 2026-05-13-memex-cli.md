# PRD: memex CLI

**Date:** 2026-05-13
**Status:** Proposed
**Branch:** main
**Author:** Rona (tech@tghtech.com)
**Effort:** ~6–10 CC-hours (~2–3 human-days) through Phase 2; Phase 3+ optional

---

## 1. One-line summary

A generic CLI + MCP server that indexes one or many Obsidian-shaped markdown vaults into Supabase + pgvector and serves hybrid (vector + keyword + RRF + multi-query + 4-layer dedup) queries — markdown stays the source of truth, vaults are first-class **sources** scoped per query via `--source`, the backlink graph is built strictly from `[[wikilinks]]`, and a new teammate can `memex clone <source>` to materialize the full vault locally and open it in Obsidian (graph view works out of the box).

---

## 2. Problem

### 2.1 What devs / agents experience today

- The e-commerce wiki is the single engineering-memory source (per `[[decisions/wiki-as-engineering-memory-source-of-truth]]`), but Claude only reaches it via raw `Grep`/`Read`/`Glob`. That's keyword-only, no ranking, no semantic recall.
- `Grep "atomic operators"` misses `[[decisions/cart-atomic-operators-not-read-mutate-save]]` if the agent searched for "atomic mongo writes" or "no read-modify-save."
- No graph traversal of `[[wikilinks]]`. To answer "what cites `concepts/lazy-upsert`?" the agent has to grep every file.
- **No multi-project memory.** A second engineering project (say, a mobile app, a data pipeline, a marketing site) gets its own Obsidian vault. Today there is no way to query across vaults *or* scope a query to one vault without juggling working directories.
- **No new-developer onboarding path.** A new teammate joining the project has to git-clone a vault repo, hope it's been pushed recently, and configure Obsidian by hand. There is no single command to materialize "the current state of the team's brain" on a fresh laptop.
- No citation auditing. Pages may reference `[Source: ...]` in inconsistent shapes and there is no tool that finds drift.
- No structural maintenance (broken wikilinks, orphan pages, untyped frontmatter) — the index gets stale silently.

### 2.2 Why the current architecture can't fix this

- `Grep`/`Read`/`Glob` are single-directory, general-purpose. They don't understand frontmatter `type`, the `[[wikilink]]` graph, H2 chunk anchors, or "which vault did this come from."
- A pure-grep approach degrades as any single vault grows past ~100 pages, and breaks entirely once there are 2+ vaults to consult.
- Adding a maintenance script per concern without a shared index means re-walking every vault N times.

### 2.3 Why this is worth fixing now

- Vault crossed 35 curated pages in ~7 days. At this growth rate it hits 200 by end of Q3 — the size where keyword-only retrieval starts missing relevant decisions.
- We already know a second project (frontend, mobile, infra, …) will get its own wiki. Designing for one vault now means rebuilding the moment the second appears.
- Every Claude Code session now starts with 3–8 `Grep` calls. A single `memex_query` MCP call returning ranked, source-aware results cuts the per-prompt overhead.

## 3. CEO review — premise challenge + alternatives

### 3.1 Premises

| # | Premise | Risk if wrong |
|---|---|---|
| P1 | Markdown stays sovereign across all vaults; the DB is derived cache | If we treat DB as truth, Obsidian breaks; vendor lock-in to Supabase |
| P2 | Any single vault will grow past ~150 pages where keyword-only retrieval starts to miss | If false, embeddings are wasted spend; FTS alone would have sufficed |
| P3 | Multi-project / multi-vault is real and imminent | If false, we paid a small modeling tax (`source_id` columns) for nothing |
| P4 | Most queries want one source; some want federation across all sources | If users always want federation, the `--source` flag becomes noise — flip the default |
| P5 | Backlinks should reflect ONLY explicit `[[wikilinks]]`, not body text mentions | If users expect "mentions" to count, recall feels low; mitigate via a separate `mentions` view in P5 |
| P6 | Supabase session pooler at install time is acceptable friction | If users won't paste a connection string, install bounces |

### 3.2 Alternatives considered

| Approach | Completeness | Effort | Verdict |
|---|---|---|---|
| A: keep Grep-only, single vault | 3/10 | 0 | Rejected — degrades with growth; can't span vaults |
| B: SQLite + FTS5 + sqlite-vec, single-vault | 6/10 | low | Rejected — no multi-source; no multi-device sync |
| C: Supabase + pgvector, single-vault | 8/10 | medium | Rejected — leaves multi-project gap for a small extra cost |
| **D (chosen): Supabase + pgvector, multi-source, `--source` flag for scoping** | 9/10 | medium | Chosen — generic, future-proof, matches direction |
| E: full gbrain fork | 10/10 | very high | Deferred — overkill; we keep ~30% of gbrain's surface |
| F: external SaaS (Notion AI, Mem.ai) | 6/10 | n/a | Rejected — breaks markdown sovereignty |

### 3.3 Scope decision

**In scope (Phase 1–4):**

- Generic vault walker + parser (frontmatter, wikilinks, H2 chunks). No e-commerce-wiki-specific code paths.
- Supabase schema with **`sources` table** (shared metadata, no per-host paths) and a `source_id` foreign key on `pages`, `chunks`, `links`, `tags`. Per-host mount paths live in **local config** so the same source can be cloned to different paths on different machines.
- Multi-vault registration: `memex sources add <name> --path <vault-dir>`; many sources per install.
- **Vault clone:** `memex clone <source> --to <local-path>` reconstructs the full vault as markdown on disk from Supabase, sets the host-local mount, scaffolds an `.obsidian/` config so Obsidian graph view works immediately. One command for new-developer onboarding.
- **Vault export:** `memex export <source> --to <local-path>` is the same content reconstruction without registering or scaffolding Obsidian config — for one-off dumps.
- Per-source sync: `memex sync --source <name>` (or all sources by default).
- `--source <name>` flag on `query`, `search`, `list`, `backlinks`, `graph-query`. Omitted = federated across all registered sources.
- Backlinks built **exclusively** from `[[wikilink]]` edges in the body of source `.md` files. No NLP, no fuzzy matching, no "mentions."
- Install-time prompt for Supabase session pooler URL, persisted in `~/.memex/config.json`.
- Hybrid query: vector (pgvector cosine) + keyword (Postgres tsvector) + RRF fusion + multi-query expansion + 4-layer dedup.
- CLI: `init`, `sources add|remove|list`, `clone`, `export`, `sync`, `query`, `search`, `get`, `list`, `backlinks`, `graph-query`, `doctor`, `skill`, `serve`.
- MCP stdio server with the read/query operations, all accepting an optional `source` parameter. (`clone`/`export` are CLI-only — onboarding is a human action.)
- 3 generic skills: `maintain`, `repo-architecture`, `citation-fixer`. Each scopes to a source.
- Citation format spec (section 7) — generic shapes, optional per-source overrides.

**Deferred to later phases:**

- HTTP server + admin dashboard
- OAuth / multi-tenant
- LLM fact extraction (gbrain's "hot memory")
- Job queue / minions
- Auto-skill resolver
- `memex ingest-url` (Playwright headless ingestion for external sources) — separate PRD
- "Mentions" graph view (body-text references that aren't wikilinks)

### 3.4 Dream state delta

- **TODAY:** Claude greps one vault 3–8 times per prompt; humans manage citations by eye; no graph queries; no multi-project context; new dev onboarding requires git-cloning a vault repo and praying it's fresh.
- **THIS PLAN:** One `memex_query` call returns ranked, cited, deduped results from one source or all; `memex doctor --source <x>` reports broken links/orphans per vault; `memex citation-fixer --source <x>` audits drift; `memex backlinks --source <x>` traverses the wikilink graph; a new teammate runs `memex clone e-commerce --to ~/code/e-commerce-wiki` and opens the materialized vault directly in Obsidian.
- **12-MONTH IDEAL:** 5+ engineering vaults registered (backend, frontend, mobile, infra, design-system). Claude pulls cross-domain context in one federated call. The CLI is a single binary the team installs via one `curl | bash`. Adding a new vault is `memex sources add <name> --path <dir> && memex sync --source <name>`. Joining the team is `memex init && memex clone <source>` per vault — vault-on-disk + Obsidian-ready, day one.

## 4. Goals & non-goals

### 4.1 Goals

- Markdown remains canonical for every registered source.
- One CLI/MCP install can manage **N vaults**; a query is either source-scoped or federated.
- Hybrid search beats raw Grep on recall@5 for a held-out set of 20 representative engineering questions, per source.
- One MCP call replaces 3–8 grep calls per Claude prompt for wiki-first lookups.
- Backlinks are deterministic, derived 1:1 from `[[wikilinks]]`, and stable across sync runs.
- Citation drift across any source is auditable and auto-fixable.
- A fresh machine with no prior vault state can become productive in two commands: `memex init` then `memex clone <source> --to <path>`. The result opens cleanly in Obsidian with graph view working.
- The roundtrip `sync → clone` is **lossless** for every page that contains only standard frontmatter + body + wikilinks. (Obsidian-specific attachments and the `.obsidian/` config dir are handled separately — see section 6.5.)

### 4.2 Non-goals

- Replacing Obsidian for humans.
- Real-time sync (manual `sync` or `--watch` is fine).
- Multi-user permissions, OAuth, audit logs.
- Replacing or competing with gbrain — this is a vault-native subset.
- Cross-source link resolution by default (`[[foo]]` in source A does NOT resolve to a page in source B unless explicitly qualified — see section 6.4).
- Storing PII or secrets — vault content only.
- Fuzzy / NLP-derived backlinks. Only explicit wikilinks count.

### 4.3 Success metrics

| Metric | Baseline | Target | Measurement |
|---|---|---|---|
| Per-prompt grep calls (Claude) | 3–8 | 0–1 | Manual session inspection over 10 prompts |
| Recall@5 on 20-question eval set per source | n/a | ≥ 85% | `memex eval --source <name> --qrels ...` |
| `sync --source <name>` time (incremental, no changes) | n/a | < 2s | `time memex sync --source <name>` |
| `sync --source <name>` time (full rebuild, ~35 pages) | n/a | < 30s | `time memex sync --source <name> --full` |
| `clone <source> --to <path>` time (~35 pages, fresh machine) | n/a | < 10s | `time memex clone ...` |
| Roundtrip fidelity: `clone` output byte-equal to original markdown for a no-op page | n/a | 100% | `diff -r` after `sync` then `clone` |
| `query` p95 latency (single source) | n/a | < 1.5s | Built-in CLI timing |
| `query` p95 latency (federated across 5 sources) | n/a | < 2.5s | Built-in CLI timing |
| Broken wikilinks detected per source | unknown | 0 | `memex doctor --source <name>` |
| Backlink count delta between consecutive no-change syncs | n/a | 0 | sanity check after `sync` |

## 5. Target user flows

### 5.1 Flow A — Install

```
$ npm install -g memex     (or curl | bash for compiled binary)
$ memex init
? Supabase pooler URL (either port — we'll derive both):
    postgresql://postgres.xxx:****@aws-0-eu-west-1.pooler.supabase.com:5432/postgres
? Embedding provider: (openai / xenova-local)        [openai]
? OPENAI_API_KEY: ****
✔ Detected session pooler on :5432; derived transaction pooler on :6543
✔ Probed both endpoints — connections OK
✔ Connected to Supabase
✔ Ran migrations (sources, pages, pages_conflicts, chunks, embeddings, links, tags)
✔ Config written to ~/.memex/config.json
✔ Add a source with: memex sources add <name> --path <vault-dir>
```

One URL is enough. CLI commands use port 6543 (transaction pooler: one PgBouncer connection per query, ideal for short-lived invocations). `memex serve --mcp` uses port 5432 (session pooler: long-lived connection for the MCP process). `init` derives one URL from the other by swapping the port and probes both endpoints; if probing fails, it falls back to asking the user for the other URL explicitly.

## 6. Data model

### 6.1 Supabase schema

```sql
-- sources: one row per registered vault (shared across all machines)
-- NOTE: no `path` column here — vault path is per-host (different on Alice's mac vs Bob's laptop).
-- Per-host paths live in local ~/.memex/mounts.json. See section 13.
create table sources (
  id           bigserial primary key,
  name         text unique not null,         -- "e-commerce", "mobile-app", ...
  description  text,
  created_at   timestamptz default now(),
  last_sync    timestamptz,                  -- most recent sync from any host
  last_sync_host text,                       -- which host did it
  config       jsonb not null default '{}'   -- per-source overrides (citation rules, types, etc)
);

-- pages: one row per .md file under a source
-- Frontmatter fields used for indexed filtering are extracted into typed columns
-- (status, author, date, supersedes_slug, superseded_by_slug); everything else
-- stays in `frontmatter` JSONB for source-specific extensions.
--
-- Precedence after sync: typed columns are the source of truth. Sync strips
-- extracted keys from the JSONB blob so they cannot drift. Only unknown keys
-- (per-source extensions, frontmatter not modeled by the schema) survive in
-- `frontmatter`. `revision_history` is written ONLY by sync (from the source
-- file) and the `revision` skill (which writes the file then triggers sync);
-- treat the DB column as read-only-after-sync — no direct UPDATEs.
create table pages (
  id                 bigserial primary key,
  source_id          bigint not null references sources(id) on delete cascade,
  path               text not null,                -- relative to source.path, e.g. "wiki/decisions/cart-singleton-keyed-by-authid.md"
  slug               text not null,                -- "decisions/cart-singleton-keyed-by-authid"
  type               text not null,                -- decision | flow | bug | concept | feature | session | other
  title              text,
  status             text not null default 'active', -- active | proposed | superseded | deprecated
  author             text,                         -- primary decider, e.g. "rona"
  co_authors         jsonb not null default '[]',  -- list of names
  date               date,                         -- frontmatter `date:` field
  supersedes_slug    text,                         -- this page replaces <supersedes_slug>
  superseded_by_slug text,                         -- this page is replaced by <superseded_by_slug>
  revision_history   jsonb not null default '[]',  -- append-only list of meaningful changes
  frontmatter        jsonb not null default '{}',  -- everything not extracted above
  body               text not null,
  body_tsv           tsvector generated always as (to_tsvector('english', coalesce(title,'') || ' ' || body)) stored,
  mtime              timestamptz not null,
  hash               text not null,                -- sha256 of file content
  content_version    bigint not null default 1,    -- bumped on every accepted write; used for concurrent-edit conflict detection
  is_raw             boolean not null default false,
  created_at         timestamptz default now(),
  updated_at         timestamptz default now(),
  unique (source_id, path),
  unique (source_id, slug)
);
create index pages_source_type_idx     on pages(source_id, type);
create index pages_source_status_idx   on pages(source_id, status);
create index pages_source_author_idx   on pages(source_id, author);
create index pages_body_tsv_idx        on pages using gin(body_tsv);

-- pages_conflicts: written when a sync's content_version compare-and-set fails.
-- Surfaces to `memex doctor`; never auto-resolved. Conflicts are MADE VISIBLE,
-- not merged. Resolution is a human decision. Conflict records OUTLIVE the page
-- they describe (forensic evidence): `page_id` is nullable so `sync --full` (which
-- truncates pages for the source) does not erase conflict history. `page_path`
-- is a denormalized snapshot of the path at conflict time so the row stays
-- interpretable after the page row goes away.
create table pages_conflicts (
  id                  bigserial primary key,
  source_id           bigint not null references sources(id) on delete cascade,
  page_id             bigint references pages(id) on delete set null,
  page_path           text not null,                -- denormalized snapshot — survives page deletion
  detected_at         timestamptz not null default now(),
  detecting_host      text not null,                -- host that tried to sync and lost the CAS
  base_content_version bigint,                      -- the version the loser was based on; null if cache-loss fallback couldn't infer one
  current_content_version bigint not null,          -- the version actually in the DB at sync time
  loser_body          text not null,                -- the body that lost the CAS — preserved for human resolution
  loser_frontmatter   jsonb not null default '{}',
  cause               text not null default 'concurrent_edit', -- concurrent_edit | cache_loss_fallback | unknown
  resolved_at         timestamptz,                  -- null until human marks it resolved
  resolution_note     text
);
create index pages_conflicts_source_open_idx on pages_conflicts(source_id) where resolved_at is null;

-- chunks: H2-split chunks for embedding
create table chunks (
  id           bigserial primary key,
  source_id    bigint not null references sources(id) on delete cascade,
  page_id      bigint not null references pages(id) on delete cascade,
  heading      text,
  ordinal      int not null,
  content      text not null,
  content_hash text not null,
  embedding    vector(1536),
  created_at   timestamptz default now()
);
create index chunks_embedding_idx on chunks using hnsw (embedding vector_cosine_ops);
create index chunks_source_idx on chunks(source_id);
create index chunks_page_idx on chunks(page_id);

-- links: edges of the graph
-- kind='wikilink' is the body-derived backlink graph (see 6.3).
-- kind='related' is the frontmatter-derived `related: [[X]]` graph — authored explicitly
-- by humans, weaker than wikilink but still semantically meaningful.
create table links (
  id           bigserial primary key,
  source_id    bigint not null references sources(id) on delete cascade,
  from_page    bigint not null references pages(id) on delete cascade,
  to_slug      text not null,                -- target slug as written in [[...]]
  to_page      bigint references pages(id) on delete set null,
  kind         text not null default 'wikilink',  -- wikilink | related
  resolved     boolean generated always as (to_page is not null) stored
);
create index links_source_idx       on links(source_id);
create index links_from_idx         on links(from_page);
create index links_to_idx           on links(to_page);
create index links_source_kind_idx  on links(source_id, kind);

-- tags: from frontmatter `tags: [...]`
create table tags (
  source_id    bigint not null references sources(id) on delete cascade,
  page_id      bigint not null references pages(id) on delete cascade,
  tag          text not null,
  primary key (source_id, page_id, tag)
);

-- sync_state: last successful sync per source per host
create table sync_state (
  source_id    bigint not null references sources(id) on delete cascade,
  host         text not null,
  last_sync    timestamptz not null,
  pages_count  int not null,
  primary key (source_id, host)
);
```

### 6.2 Why Postgres tsvector + pgvector together

- `tsvector` gives proper stemming, stop-word handling, and `websearch_to_tsquery`.
- `pgvector` with `hnsw` gives sub-100ms cosine search up to ~100k rows.
- Both live in the same DB; one query plan can join them and filter by `source_id`.
- Supabase session pooler keeps connection overhead amortized.

### 6.3 Backlinks: wikilink-only, by spec

The backlink graph is derived **exclusively** from `[[wikilink]]` occurrences in page bodies. Specifically:

- During parse: extract every `[[target]]` and `[[target|alias]]` pattern from the page body.
- Insert one `links` row per occurrence: `(source_id, from_page, to_slug=target)`.
- After all pages are parsed, resolve `to_slug` → `to_page` within the same `source_id` (no cross-source resolution by default).
- `memex backlinks <slug>` answers: `select from_page from links where to_page = <slug page id> and source_id = <scope>`.
- An unresolved link (`to_page is null` after sync) is reported by `doctor` as a broken wikilink.

What does NOT count as a backlink:

- Body text that contains the slug as plain words ("we use lazy-upsert here")
- Markdown link `[text](path)` syntax
- HTML anchors
- `[Source: ...]` citations (these are tracked separately by `citation-fixer`; they do not produce wikilink edges)

This is a deliberate design choice (see premise P5 and section 17 decision 5). A future "mentions" view can be added without breaking the wikilink graph contract.

### 6.4 Cross-source links

By default, `[[some-slug]]` resolves only within the same source. To intentionally cross sources, use the qualified form `[[source-name:slug]]` (e.g. `[[mobile-app:decisions/biometric-refresh]]`). The walker recognizes this prefix and resolves `to_page` across sources. Unqualified cross-source intent is treated as ambiguous and flagged by `doctor`.

### 6.5 Vault reconstruction (clone / export)

`pages.body` stores the raw markdown body; `pages.frontmatter` stores parsed YAML as JSONB. To reconstruct an `.md` file on disk:

1. Serialize `frontmatter` → YAML (canonical key order: `type`, `date`, `tags`, `related`, then the rest alphabetically).
2. Emit `---\n<yaml>\n---\n\n<body>`.
3. Write to `<target-path>/<pages.path>` (preserving the subdirectory structure exactly as stored).
4. Set the file mtime to `pages.mtime` so a subsequent `memex sync` correctly sees the page as unchanged.
5. After all files: write a minimal `.obsidian/` folder (`app.json`, `workspace.json`, `graph.json`) so Obsidian opens the vault with sensible defaults — including the graph view enabled. `clone` does this; `export` does not.

**Roundtrip fidelity guarantee:** for any page whose body uses only standard markdown + frontmatter + `[[wikilinks]]`, `sync → clone` produces a byte-equal file (modulo YAML key ordering, which we normalize at sync time so both ends agree).

**What's NOT carried in `pages.body`:**

- Binary attachments (images, PDFs, audio) referenced via `![[...]]` — these need a separate object store. v1 stores them only by reference; v2 adds Supabase Storage backing.
- `.obsidian/` plugin configs and per-user settings. `clone` writes only the minimum default; users carry their own plugin prefs.
- Files outside `wiki/` and `raw/` (e.g. `.gitignore`, scripts). These belong to whoever owns the underlying git repo, not the brain.

**Conflict handling on clone:**

- Default: refuse if `<target-path>` exists and contains `.md` files.
- `--force`: overwrite. Used for re-baseline after divergence.
- `--merge` (P5): three-way merge against local edits. Out of scope for v1.

## 7. Citation + contradiction model

Designed to extend what the e-commerce vault already does (frontmatter + dense `[[wikilinks]]`) rather than impose a foreign shape. Three layers: page-level provenance, inline references, inline amendments. Plus an explicit supersession DAG for handling decision reversals.

### 7.1 Layer 1 — Page-level provenance (frontmatter)

Every `wiki/` page carries:

```yaml
type: decision                 # decision | flow | bug | concept | feature | session
date: 2026-05-12
status: active                 # active | proposed | superseded | deprecated
author: rona                   # primary decider — single name
co_authors: [claude]           # drafters/proposers — optional list
tags: [...]
sources: [raw/features/..., raw/sessions/...]   # the raw inputs this page distills
supersedes: [[decisions/old-slug]]       # set when this page replaces another
superseded_by: [[decisions/new-slug]]    # set when status == superseded
revision_history:              # append-only, meaningful changes only
  - { date: 2026-05-12, author: rona, change: initial }
  - { date: 2026-05-13, author: rona, change: raised threshold 500ms → 2s after [[bugs/slow-net]] }
```

`author` and `co_authors` are single names (`rona`, `claude`). Solo-shop scale; no full names or role tags. `revision_history` records meaningful changes only — typos and formatting belong to git, not the wiki.

### 7.2 Layer 2 — Inline references (body)

`[[wikilinks]]` are the citation primitive. No `[Source: ...]` wrapper — the wikilink IS the citation when used as a body reference. Three thin extras only where the target isn't a wiki page:

| Kind | Shape | Example |
|---|---|---|
| Wiki page (same source) | `[[wiki/path/slug]]` or `[[concepts/lazy-upsert]]` | `…uses [[concepts/lazy-upsert]] for race-free upsert.` |
| Wiki page (other source) | `[[source-name:slug]]` | `…matches [[mobile-app:decisions/biometric-refresh]].` |
| Source code | `path:line` | `…enforced at src/services/cart.service.js:42.` |
| Git commit | `commit <hash>` | `…fixed in commit abc1234.` |
| External URL | bare URL | `…per https://datatracker.ietf.org/doc/html/rfc6749.` |
| Claude inference (raw only) | `[Inferred · YYYY-MM-DD]` | Never allowed in `wiki/`. Allowed in `raw/sessions/`. |

### 7.3 Layer 3 — Inline amendments

For tweaks that refine an existing decision without reversing it. Block lives in the body forever so readers see the change without `git blame`:

```markdown
## Decision

We use approach X.

> **Amendment 2026-05-13 — @rona:** Originally 500ms; raised to 2s after [[bugs/slow-net-false-positive]] showed legitimate slow clients tripping the threshold.
```

Every inline amendment must be paired with a one-line entry appended to `revision_history` in frontmatter so the audit trail is both human-visible and machine-queryable.

### 7.4 Contradiction handling — three cases by scope

| Scope | Action |
|---|---|
| **Full reversal** — the prior decision is now wrong | Create a new decision page with `supersedes: [[old-slug]]`. Old page: flip `status: superseded`, add `superseded_by: [[new-slug]]`, prepend a `## Superseded YYYY-MM-DD` section explaining what changed and why. Old page is **never deleted**. |
| **Mid-implementation pivot** — wiki not yet written | Just write the decision reflecting the new shape. If a PRD in `raw/features/` carries the old plan, append a `## Amendment YYYY-MM-DD` section. Don't rewrite history in `raw/`. |
| **Inline tweak** — same decision, refined detail | Use a Layer-3 amendment block + a `revision_history` entry. Same page. No new file. |

### 7.5 `citation-fixer` — contradiction-fix mode

Runs on every page in scope:

1. Build a supersession DAG from frontmatter `supersedes` ↔ `superseded_by` edges per source.
2. Verify edges are symmetric (both endpoints reference each other); flag asymmetries. **Note:** symmetry is normally an invariant of sync — see section 12, where `sync` auto-populates the inverse edge on a source page when ingesting one with `supersedes:`. `citation-fixer` is the belt to sync's suspenders; if it finds an asymmetry, something bypassed sync.
3. For each page with `status: active`, error if anything supersedes it.
4. For each `[[wikilink]]` in an active page body whose target is `superseded`: insert an adjacent marker — `[[old-slug]] (now superseded by [[new-slug]])`. **Never auto-rewrite the link itself**; the semantic intent may be historical rather than current.
5. For each code reference (`path:line`): verify the line exists at `sources.config.code_repo + path`. **Batch by file:** group all references targeting the same file path, read each file exactly once, check all referenced lines against the file's line count. One read per cited file, not one read per citation.
6. **Strict mode (CI gate):** non-zero exit if any `status: active` page cites a superseded page without the adjacent marker, or any code reference is stale.

The fixer also generates a sidecar `wiki/.history/<chain>.md` walking the supersession chain so `memex skill maintain` can render a decision-evolution timeline on demand.

### 7.6 Per-source overrides

A source can extend or relax rules via `sources.config.citation`:

```json
{
  "citation": {
    "code_repo": "../e-commerce-backend",        // root for resolving src/foo.js:42 paths
    "require_supersedes_pair": true,             // strict mode for this source
    "allow_inferred_in_wiki": false,             // default; flip true if you want to be permissive
    "extra_inline_shapes": {
      "linear_ticket": "LIN-\\d+"
    }
  }
}
```

Defaults: `code_repo` unset (code citations not verified), `require_supersedes_pair` false in audit, true in strict, `allow_inferred_in_wiki` false.

### 7.7 Audit / fix / strict — three invocation modes

| Mode | Reads | Writes | Exit code on issue |
|---|---|---|---|
| `citation-fixer --source <x> --audit` | wiki + code (if `code_repo` set) | nothing | 0 always (reports only) |
| `citation-fixer --source <x> --fix` | wiki + code | rewrites markdown for safe drift (bare URL wrap, adjacent supersession marker, missing `revision_history` entry inferred from amendment blocks) | 0 if fixed |
| `citation-fixer --source <x> --strict` | wiki + code | nothing | non-zero on any active page violating rules 1–6 |

## 8. Skills

Five skills ship with v1. Each is a markdown file under `memex/skills/<name>.md`, invocable via `memex skill <name> [--source <x>]` and (for the read-only ones) as an MCP tool. Skills are generic — they read a source's pages and act on them.

### 8.1 `maintain`

Wraps `doctor` + adds:

- Stale-info detection (page references a slug that has been renamed)
- Decision pages older than 90 days not re-linked
- Frontmatter shape validation against a per-source schema (frontmatter shapes can vary per project)
- Index/log consistency
- Open conflict count from `pages_conflicts` (per section 6.1) — surfaces unresolved concurrent-edit conflicts to the team

### 8.2 `repo-architecture`

Codifies "where new brain files go" *per source*.

- Reads `<source>/CLAUDE.md` and `<source>/APPROACH.md` (or equivalent) to learn the source's filing schema
- Input: free-text description of new content
- Output: canonical path within the source + reasoning
- Surfaces near-duplicates the user should update instead

### 8.3 `citation-fixer`

Per section 7. Audit / fix / strict modes, source-scoped.

### 8.4 `rename` — migrate a page slug

`memex skill rename <old-slug> <new-slug> --source <name>` rewrites every inbound `[[wikilink]]` in body markdown across all pages in the source to point at the new slug, then renames the file on disk, then re-syncs. Without this, every renamer has to grep-and-replace by hand and breaks references silently.

**Clean-state precondition.** Rename refuses to run unless:

1. No open rows in `pages_conflicts` for this source (resolved_at IS NULL count = 0).
2. No pages with `mtime > sources.last_sync` (i.e., no unsynced local edits — the file system matches the last sync's view).
3. No other `memex` process holds the source's sync lock (`~/.memex/cache/<source>.lock`).

If any check fails, rename bails with a clear message: "Source has 3 unsynced edits — run `memex sync --source <name>` first." This prevents the race where another teammate's concurrent edit re-creates a page at the old slug after rename has already rewritten inbound references.

Steps (after preconditions pass):

1. Confirm `<old-slug>` exists in the source; bail if not.
2. Confirm `<new-slug>` does not exist; bail if it does.
3. Take the source sync lock for the duration of the rename.
4. `select from_page from links where to_slug = <old-slug> and source_id = <source>` to enumerate citing pages.
5. For each citing page: read the file on disk, rewrite `[[<old-slug>]]` and `[[<old-slug>|<alias>]]` occurrences to the new slug, write back.
6. Rename the page file on disk: `<old-path>` → `<new-path>` (preserve directory).
7. Release the lock.
8. `memex sync --source <name>` to re-index. Sync resolves the now-current wikilinks to the new `page_id`.

Dry-run mode (`--dry-run`) reports which files would change without writing. Dry-run skips the lock and the preconditions.

### 8.5 `revision` — append a revision_history entry safely

`memex skill revision <slug> --source <name> --author <x> --change "..."` appends an entry to the page's `revision_history` frontmatter array using a proper YAML emitter. Hand-editing this array in Obsidian is fragile — broken YAML breaks the page. This helper guarantees valid output.

- Reads the page file.
- Parses frontmatter (gray-matter).
- Appends `{ date: <today>, author: <x>, change: <text> }` to `revision_history`.
- Re-emits with canonical key order.
- Writes back; sync picks up the change on next run.

Pairs naturally with the inline amendment block convention from section 7.3.

## 9. Hybrid search pipeline

```
query "<question>"  [--source <name>]
   │
   ▼
[0] source filter: scope = {source_id ∈ requested set} or {all}
   │
   ▼
[1] intent classify (regex + heuristics, no LLM)
       type-filter? tag-filter? graph-only?
   │
   ▼
[2] multi-query expansion (Haiku, 1 call, LRU-cached on original query)
       → "cart currency lock"
       → "first-item currency invariant"
       → "currency mismatch 409"
       SKIP this step if the query is federated across 2+ sources:
         federation already provides recall diversity, and the Haiku call
         pushes federated p95 past its 2.5s budget. Single-source queries
         get expansion; federated queries don't.
   │
   ▼
[3] parallel retrieval
       single-source: 3 queries × 2 retrievers = 6 result sets
       federated:     1 query  × 2 retrievers = 2 result sets
       pgvector cosine top-20  WHERE source_id IN scope  ────┐
       tsvector websearch top-20 WHERE source_id IN scope ───┤
                                                              ▼
[4] RRF fusion: score = Σ 1 / (60 + rank)
   │
   ▼
[5] cosine re-rank against the ORIGINAL query embedding
   │
   ▼
[6] backlink boost: pages with more inbound wikilinks (per scope) get +10%
   │
   ▼
[7] 4-layer dedup
       layer 1: by (source, path) — no two chunks from same file in top 3
       layer 2: by (source, slug) — handle path renames within a source
       layer 3: by chunk content hash
       layer 4: by H2 heading within a page (one "## Decision" max)
   │
   ▼
[8] return top-K with citations (source, path, heading, snippet, score)
```

When federated (no `--source`), step [6]'s backlink boost is computed per source then re-normalized so a source with denser linking doesn't trivially dominate.

## 12. Sync strategy

Per source:

- **Walk:** `find <source.path> -name '*.md'`
- **Per file:** read, hash (sha256), compare to `pages.hash`. If unchanged, skip.
- **If changed:** parse frontmatter (gray-matter), extract wikilinks via regex `\[\[([^\]\|]+)(?:\|[^\]]+)?\]\]`, chunk by H2 (split on `\n## `).
  - **Extract typed columns from frontmatter:** `status`, `author`, `co_authors`, `date`, `supersedes` → `supersedes_slug`, `superseded_by` → `superseded_by_slug`, `revision_history`. Whatever's left goes into `frontmatter` JSONB.
  - **Concurrent-edit conflict check (with cache-loss fallback).** Read the local file's recorded "based-on version" from `~/.memex/cache/sync-state.json` for this `(source, path)`. Three cases:
    1. **Based-on version known AND matches DB `content_version`:** perform atomic `UPDATE pages SET ..., content_version = content_version + 1 WHERE source_id = $1 AND path = $2 AND content_version = $3`. Row count = 1 → success, write new based-on version to cache.
    2. **Based-on version known AND differs from DB `content_version`:** real concurrent-edit conflict. Write local body + frontmatter into `pages_conflicts` with `cause='concurrent_edit'`. Skip the page. Report to user.
    3. **Based-on version MISSING from local cache** (cache file deleted, corrupted, or new clone): fall back to body-hash comparison. Compute sha256 of the local file body. If equal to the DB's `pages.hash` → the local copy IS the current DB version, no edit happened. Take the DB row's `content_version` as the new based-on, write it to the cache, skip the page. If unequal → we can't tell whether this is a fresh local edit on top of the current DB version or a real conflict; conservatively write to `pages_conflicts` with `cause='cache_loss_fallback'` and `base_content_version = NULL`. Surfaces in `doctor`; human resolves by either accepting the local edits (`memex skill resolve-conflict <id> --accept-local`) or keeping the DB version (`--accept-db`).

    The fallback turns cache deletion from "mass false-positive conflict storm" into "one extra hash compare per page on the next sync." The cache becomes an optimization, not a correctness dependency.
  - **Insert wikilinks** as `links` rows with `kind='wikilink'`, one per occurrence.
  - **Insert frontmatter `related: [[X]]`** as `links` rows with `kind='related'`, one per entry in the `related:` array.
  - Delete-and-reinsert chunks for the page; embed only new chunks.
- **After all pages in the source — batch link resolution.** Resolve `links.to_slug` → `links.to_page` in **one** SQL statement, not per row:

  ```sql
  UPDATE links l
     SET to_page = p.id
    FROM pages p
   WHERE l.source_id = $1
     AND l.to_page IS NULL
     AND p.source_id = l.source_id
     AND p.slug = l.to_slug;
  ```

  Qualified cross-source links `[[other-source:slug]]` get a separate batch UPDATE with a join across sources (per 6.4).

- **After all pages — auto-populate inverse supersession edges (hash-gated).** For every page Y with `supersedes_slug = X`, ensure page X has `superseded_by_slug = Y`. Sync writes this to the DB row unconditionally; writes it back to the source markdown file ONLY when the local file's body hash equals `pages.hash` (i.e., no unsynced local edits since last sync). This prevents fighting Obsidian's unsaved buffer: if the user has X open in Obsidian with pending changes, sync won't yank the file out from under them. When the write-back is skipped, the symmetry write is queued in `pages.frontmatter -> '_pending_symmetry_write'` and reported by `doctor` until the next sync finds the file clean and completes it. Symmetry is an invariant of sync's DB state at all times; the source-file mirror catches up on the next clean sync.

  Additionally, after extracting typed columns from frontmatter, sync re-emits the page's `frontmatter` JSONB column with extracted keys STRIPPED — only unknown keys (per-source extensions) survive there. This prevents `pages.frontmatter -> 'status'` from drifting away from `pages.status`. Typed columns win.

- **Full sync (per source):** delete `pages WHERE source_id = X` (cascades to chunks/links/tags); `pages_conflicts` is preserved (truncating it would lose conflict evidence). Rebuild pages from scratch.

`memex sync` with no `--source` walks every registered source serially. `--parallel` for fan-out is a P5 polish.

## 13. Install + config

```
~/.memex/
├── config.json        global config (Supabase URL, embed provider, defaults)
├── mounts.json        per-host mount registry: { source name → local vault path }
└── cache/             multi-query LRU, last-sync host fingerprint
```

`mounts.json` is the local "where is each vault on THIS machine" map. Populated by `memex sources add` and `memex clone`. Example:

```json
{
  "host": "bob-laptop",
  "mounts": {
    "e-commerce": "/home/bob/code/e-commerce-wiki",
    "mobile-app": "/home/bob/code/mobile-wiki"
  }
}
```

Same source, different path on every machine. This is why path lives in local config and not the `sources` table.

`config.json`:

```json
{
  "supabaseTransactionPoolerUrl": "postgresql://postgres.xxx:****@aws-0-eu-west-1.pooler.supabase.com:6543/postgres",
  "supabaseSessionPoolerUrl":     "postgresql://postgres.xxx:****@aws-0-eu-west-1.pooler.supabase.com:5432/postgres",
  "embedProvider": "openai",
  "embedModel": "text-embedding-3-small",
  "openaiApiKey": "***",
  "indexRaw": false,
  "multiQueryEnabled": true,
  "rrfK": 60,
  "defaultSource": null
}
```

**Two pooler URLs in the persisted config, one prompt at install.** `init` accepts one URL and derives the other by swapping ports (5432 ↔ 6543). Both endpoints are probed before persisting; if probing fails (e.g. a self-hosted Supabase with non-standard ports), `init` falls back to prompting for the second URL explicitly. The persisted config always carries both so runtime never has to re-derive.

Transaction pooler (port 6543) is used by every short-lived CLI command — open, query, close in milliseconds. Session pooler (port 5432) is used by `memex serve --mcp`, which holds a long-lived connection for the MCP stdio process. Mixing them up (session pooler for CLI) saturates Supabase connection limits at small team scale.

`defaultSource` lets a user set a fallback if they typically work in one vault (e.g. `defaultSource: "e-commerce"`). Explicit `--source` always overrides.

Per-source metadata lives in the DB (`sources` table), not in `config.json`, so multiple machines share the same registry via Supabase.

Secret handling: `openaiApiKey` and both pooler URLs are masked in `doctor` output. Optional `--env-file` points at `.env` instead of writing them to disk.

## 15. Open questions

1. **Embedding provider default — OpenAI or local Xenova?** OpenAI faster + better; Xenova free + offline. Default OpenAI, Xenova documented.
2. **Where does `.mcp.json` live?** Lean: each project repo points to a single global `memex` MCP server. Sources are managed centrally.
3. **Should `raw/` be indexed by default per source?** Lean NO. `--include-raw` opens it. Configurable per source via `sources.config`.
4. **Federated query result mixing — interleave or partition?** Lean interleave by score (already what RRF gives us) with a per-source cap to prevent dominance.
5. **Where does the eval qrels set live?** Per source: `<source>/eval/qrels.json` so each project owns its eval.
6. **Default frontmatter `type` taxonomy across sources?** Lean: provide a default (`decision|flow|bug|concept|feature|session`) but allow per-source extension via `sources.config.types`.

## 16. Risk register

| Risk | Likelihood | Mitigation |
|---|---|---|
| Supabase session pooler hits connection limit at team scale | medium | Split pooler: transaction pooler for CLI (port 6543), session pooler for MCP (port 5432). See section 13. |
| Two devs edit the same page on different machines → silent data loss on second sync | medium | `content_version` CAS at sync time; loser writes to `pages_conflicts` table; `doctor` surfaces. Never auto-merge — make conflicts visible. |
| Open `pages_conflicts` rows accumulate if devs ignore them | medium | `doctor` shows open count; `maintain` skill nags above a threshold; resolution requires explicit `memex skill resolve-conflict <id>` (P5) — for v1, humans edit the page and re-sync, then run `memex doctor --fix` to mark resolved |
| `text-embedding-3-small` becomes too coarse at 10k+ pages across sources | medium | Schema supports `embedding_model` column for re-embedding; bump dimension or model later |
| Markdown parsing edge cases (nested wikilinks, code fences) | medium | Use `remark` + `remark-wiki-link`; regex only for chunking |
| Drift between a vault and DB after manual edits | low | `sync --source <x> --full` recovers; documented as the rebuild path |
| MCP server crash drops queries silently | low | stdio crash propagates to Claude; structured error response with `error.code` |
| User expects fuzzy backlinks ("mentions") and is surprised by zero | medium | Document wikilink-only contract clearly; add mentions-view in P5 |
| Source name collisions on two machines | low | `sources.name` is unique; second machine importing same source gets a conflict error and resolution prompt |
| `clone` against a non-empty target wipes a user's work | low | Default refuse; `--force` is opt-in. Add a `.memex-cloned-at` marker so re-clones can detect their own previous run |
| Attachments referenced via `![[image.png]]` 404 after clone (v1 doesn't carry binaries) | medium | v1 documents this clearly; `doctor` reports `![[…]]` references with missing files; Supabase Storage backing in P5 |
| `.obsidian/` scaffold conflicts with a team's existing plugin config | low | Scaffold only if `.obsidian/` is absent; never overwrite existing config |
| Frontmatter YAML key ordering causes spurious diffs across machines | medium | Normalize key order at sync time so `clone` output is deterministic |
| `revision_history` YAML corruption from hand editing | medium | `memex skill revision` appends entries with a proper YAML emitter; hand-edit remains possible but no longer default |
| `supersedes`/`superseded_by` asymmetry breaks the supersession DAG | low | Sync auto-populates the inverse edge in both DB and source markdown; `citation-fixer` is belt-and-suspenders |
| Code-citation verification reads files redundantly | low | `citation-fixer` batches by file path — one read per cited file, not per citation |
| Federated query exceeds 2.5s p95 latency budget | medium | Skip multi-query expansion when federated (per section 9 step 2); federation already provides recall diversity |
| **Cache loss = mass false-positive conflicts** (delete `~/.memex/cache/` → every page trips the CAS) | medium | Body-hash fallback: when local based-on version is missing, compare local file hash against `pages.hash`. Equal = no conflict. Cache is an optimization, not a correctness dependency. See section 12. |
| **Auto-write-back of inverse supersedes fights Obsidian's unsaved buffer** | medium | Hash-gate the write: only mirror to source file when local hash equals `pages.hash`. Otherwise queue in `frontmatter._pending_symmetry_write`; `doctor` reports. Symmetry invariant lives in DB; file mirror catches up on next clean sync. |
| Typed frontmatter columns drift from JSONB blob copy of same key | low | Sync strips extracted keys from JSONB on every run. JSONB only carries unknown keys. Documented in section 6.1. |
| `pages_conflicts` evidence lost when `sync --full` truncates pages | low | `page_id ON DELETE SET NULL` + denormalized `page_path` so rows survive page deletion. |
| `graph-query --depth N` runs unbounded recursion on dense graph | low | Default depth 3, hard cap 10, error above. CLI help text says so. |
| Rename racy with concurrent editor on the old slug | medium | `rename` requires clean state (zero open conflicts, zero unsynced edits, sync lock held) before rewriting. Documented in section 8.4. |

## 17. Decisions committed (to be promoted to `wiki/decisions/` on ingest)

1. **memex is a generic multi-source CLI on top of Obsidian-shaped vaults.** Markdown is sovereign; Supabase + pgvector is the derived index. Amends `[[decisions/wiki-as-engineering-memory-source-of-truth]]`.
2. **Supabase session pooler over local PGLite.** Hosted, multi-device-ready, multi-source-ready.
3. **Sources are first-class.** `sources` table; every page/chunk/link/tag carries `source_id`. `--source` flag scopes reads. Omitted = federated.
4. **Cross-source links use qualified syntax `[[source:slug]]`.** Default resolution is intra-source.
5. **Backlinks are wikilink-only.** `[[target]]` is the sole edge generator. No NLP, no fuzzy matching, no body-text mentions. Future "mentions" view is additive, not retroactive.
6. **Hybrid search with multi-query expansion + 4-layer dedup.** Beats keyword-only at vault scale > 100 pages; Haiku expansion call is LRU-cached.
7. **Citation format codified with per-source overrides.** Defaults in section 7; sources can extend via `sources.config.citation`.
8. **MCP is the agent path; CLI is the human path.** Same engine, two front ends. Every read operation accepts an optional `source`.
9. **Vault reconstruction is built-in (`clone` / `export`).** Supabase doubles as a vault hub: any teammate can materialize the current state of a vault on a fresh machine with one command. `clone` also writes an `.obsidian/` scaffold so the graph view works out of the box.
10. **Source path is per-host, not per-vault.** The `sources` table holds shared identity and config; local `~/.memex/mounts.json` holds this machine's path. Same source clones to different paths on Alice's and Bob's machines without any DB churn.
11. **Lossless roundtrip is a contract.** A page that uses only standard frontmatter + body + wikilinks is byte-equal after `sync → clone` (with frontmatter key order normalized at sync time). Attachments and `.obsidian/` plugin configs are out of scope for the contract.
12. **Dual Supabase pooler URLs.** Transaction pooler (6543) for short-lived CLI commands; session pooler (5432) for the long-lived MCP server. Same project, different ports. Mixing them up saturates Supabase connection limits at team scale.
13. **Concurrent edits surface as conflicts, never auto-merge.** `pages.content_version` is bumped on every accepted write; sync uses CAS to detect concurrent edits and writes losers to `pages_conflicts`. `doctor` reports the open conflict count. Make conflicts visible — let humans resolve them.
14. **Supersession DAG symmetry is enforced at sync, not at audit.** When sync ingests page Y with `supersedes: [[X]]`, it writes `superseded_by: [[Y]]` back into X's frontmatter AND its DB row. `citation-fixer` checks for asymmetry as belt-and-suspenders, but the invariant lives in sync.
15. **`related: [[X]]` is a first-class edge.** Modeled in `links` with `kind='related'`, distinct from `kind='wikilink'`. Authored explicitly in frontmatter; not body-derived. Weaker than wikilink but still queryable via `graph-query`.
16. **Distribution is P1, not optional.** `bun build --compile` ships a single binary at the end of P1 so the second teammate can install it on day one — the new-dev clone flow doesn't work without it.
17. **Cache is an optimization, not a correctness dependency.** When the local cache lacks a based-on version for a page, sync falls back to body-hash comparison against `pages.hash`. Equal = no conflict. Different = surface as `cause='cache_loss_fallback'` conflict for human resolution. Cache deletion never produces silent data loss or a mass false-positive conflict storm.
18. **Source-file writes are hash-gated.** Sync only writes back to a `.md` file (e.g., for inverse-supersedes mirror) when the file's on-disk hash matches `pages.hash`. Pending writes are queued in frontmatter and reported by `doctor`. The tool never fights Obsidian's unsaved buffer.
19. **Typed columns win over JSONB.** After sync, extracted frontmatter keys live ONLY in their typed columns; the JSONB blob carries only unknown keys. No two-source-of-truth drift.
20. **Conflict evidence outlives the page row.** `pages_conflicts.page_id` is nullable; `page_path` is denormalized. `sync --full` (which truncates pages) preserves forensic conflict history.
21. **Install asks for one URL, persists two.** `init` derives transaction-pooler URL from session-pooler URL by port swap (or vice versa) and probes both before persisting. Fallback to explicit dual-prompt only when probing fails (self-hosted Supabase).

- [ ] `memex clone e-commerce --to ~/tmp/fresh-vault` materializes all 35 pages in < 10s on a host that has never touched the source
- [ ] After clone, opening `~/tmp/fresh-vault` in Obsidian shows the full graph view with all wikilink edges, no manual config
- [ ] Roundtrip fidelity: `memex sync` on the cloned vault produces zero changes (mtimes restored, body byte-equal)
- [ ] `memex clone` refuses to overwrite a non-empty target by default; `--force` overrides
- [ ] `memex export <source> --to <path>` produces the same markdown but writes no `.obsidian/` and does not register the source
- [ ] `~/.memex/mounts.json` is updated after `clone`; `sources list` reflects the new host-local path
- [ ] Concurrent edit detection: simulate two hosts editing the same page; the second `sync` writes the losing version to `pages_conflicts` and `doctor` reports it. The DB page is NOT silently overwritten.
- [ ] Supersession symmetry: writing a new decision page Y with `supersedes: [[X]]` and running `sync` causes X's source markdown file AND DB row to gain `superseded_by: [[Y]]`. `citation-fixer` finds zero asymmetries.
- [ ] `memex skill rename old-slug new-slug --source e-commerce` rewrites every inbound `[[old-slug]]` in body markdown across the source, renames the file, and re-sync resolves links to the new page. `--dry-run` reports the diff without writing.
- [ ] `memex skill revision <slug> --source e-commerce --author rona --change "raised threshold"` appends a valid YAML entry to `revision_history` with today's date.
- [ ] `related: [[X]]` in a page's frontmatter is modeled as a `kind='related'` row in `links` after sync; `graph-query <slug> --depth 1` returns both wikilink and related neighbors.
- [ ] `citation-fixer` with code-citation verification reads each referenced file at most once per run (batched by path), even when many citations target the same file.
- [ ] Federated query (no `--source`) does NOT trigger multi-query expansion; single-source query DOES. Verifiable from internal telemetry / debug output.
- [ ] `bun build --compile` produces a single-file binary at the end of P1; `./memex --version` works on a host that has no Node installed.
- [ ] **Cache-loss recovery (issue 1A fix):** delete `~/.memex/cache/`, run `sync --source e-commerce` on an unchanged vault, observe zero `pages_conflicts` rows written (hash-fallback recognizes local == DB). Then modify a single page locally, delete cache again, sync — expect exactly one `pages_conflicts` row with `cause='cache_loss_fallback'` for that page.
- [ ] **Auto-write-back gating (issue 1B fix):** edit a superseded page in Obsidian (unsaved buffer); run `sync` for a different page that triggers an inverse-supersedes mirror to the edited file; observe Obsidian gets NO "file changed externally" prompt; the symmetry write is queued in `frontmatter._pending_symmetry_write` and reported by `doctor`. After saving in Obsidian and re-syncing, the queued write completes and the marker disappears.
- [ ] **Conflict resolution path:** after a `pages_conflicts` row exists, human edits the source page to incorporate both versions and runs `sync`. The row's `resolved_at` is set; `doctor` no longer reports it.
- [ ] **Related-vs-wikilink isolation:** a page with `related: [[X]]` in frontmatter and no `[[X]]` in body causes `memex backlinks X` to NOT include this page; `memex graph-query X` DOES include it (as a `kind='related'` edge).
- [ ] **Typed-column precedence:** after sync, `pages.frontmatter` JSONB does NOT contain the keys `status`, `author`, `co_authors`, `date`, `supersedes`, `superseded_by`, `revision_history` (they live in typed columns). Only unknown keys survive in JSONB.
- [ ] **`pages_conflicts` survives `sync --full`:** insert a conflict row, run `sync --source <x> --full` (which deletes all pages for the source), observe the conflict row still exists with `page_id IS NULL` and `page_path` intact for forensics.
- [ ] **`rename` clean-state precondition:** with an unsynced edit pending, `memex skill rename old new --source <x>` bails with a clear error; after `sync`, rename succeeds.
- [ ] **`graph-query` depth cap:** `graph-query <slug> --depth 11` errors out; `--depth 10` works.
- [ ] **`init` single-URL flow:** paste only the session-pooler URL (`:5432`); `init` derives and probes `:6543`, persists both, completes without a second prompt.
- [ ] Recall@5 ≥ 85% on the per-source eval qrels set
